## 📌 Summary
`Stream API`와 전통적인 `For-loop`의 성능을 비교 분석하고, 데이터 가공 시 상황별 최적의 반복 방식 선택 기준을 수립하였습니다.

## ✍️ Description
10,000건의 사용자 이름 데이터를 대상으로 특정 문자열(`"alice"`) 필터링 작업을 수행하여 세 가지 방식의 처리 시간을 측정하였습니다.

**성능 측정 로직 분석**:
  - **For-loop**: 별도의 객체 생성 없이 리스트에 직접 접근하여 처리.
  - **Sequential Stream**: 선언적 필터링을 통해 코드 가독성을 확보함.
  - **Parallel Stream**: `ForkJoinPool`을 활용한 멀티코어 분산 처리를 시도함.



**[Troubleshooting]**
`streamTest()` 과정에서 스트림 객체 자체를 출력할 경우 데이터 내용이 아닌 객체 주소값이 출력되는 현상을 확인하였습니다. 이를 통해 스트림은 최종 연산(`collect`, `forEach` 등)이 호출되어야만 실제 데이터 처리가 시작된다는 점을 명확히 하였습니다.

## 💡 PR Point
테스트 결과 도출된 핵심 인사이트입니다.

1. **오버헤드 발생 확인**:
   - 10,000건 규모의 데이터에서는 `Parallel Stream`의 스레드 분할 및 병합 비용(Overhead)이 실제 연산 이득보다 커서 `Sequential`보다 성능이 낮게 측정될 수 있습니다.
2. **단순 연산의 효율성**:
   - 단순한 문자열 비교(`startsWith`) 작업에서는 부가적인 객체 생성이 없는 `For-loop`가 가장 빠른 응답성을 보였습니다.
3. **가독성 vs 성능의 트레이드오프**:
   - 성능 차이가 미미한(ms 단위 이하) 경우, 코드의 유지보수성과 가독성을 위해 `Sequential Stream` 사용이 권장됩니다.
